สรุปความหมายของ quota แต่ละประเภทและวิธีอ่านค่า เพื่อช่วยตั้ง policy ได้ไม่สับสน

Thai Guide
Token quota

จำกัดจำนวน token ที่ผู้ใช้ใช้ได้ในช่วง daily/monthly เหมาะกับการควบคุมปริมาณการใช้งานโดยรวม ค่า 0 หมายถึงไม่จำกัดในช่วงนั้น

Cost quota

จำกัดค่าใช้จ่ายจริง/ประมาณการเป็น USD เหมาะกับการคุมงบประมาณ เพราะโมเดลแต่ละตัวมีราคาต่อ token ไม่เท่ากัน ค่า 0 หมายถึงไม่จำกัด

Formula quota

ใช้ token และ cost ร่วมกันเป็นคะแนนเดียว เหมาะกับ policy ที่ต้องคุมทั้งปริมาณและงบพร้อมกัน เช่น จำกัดไม่ให้ใช้ token เยอะหรือใช้โมเดลแพงจนเกินเพดาน

Formula calculation

1) คำนวณ ratio: token_ratio = token_used / token_limit และ cost_ratio = cost_used / cost_limit. ถ้า limit ของฝั่งใดเป็น 0 ฝั่งนั้นจะไม่ถูกนำมาคิดเป็นตัวบล็อก

2) Max Ratio: ใช้คะแนน score = max(token_ratio, cost_ratio). ถ้า token หรือ cost ฝั่งใดฝั่งหนึ่งแตะ threshold ก็ block ทันที เหมาะกับ policy แบบ “ห้ามเกินเพดานด้านใดด้านหนึ่ง”

3) Weighted Ratio: ใช้คะแนน score = token_ratio × token_weight + cost_ratio × cost_weight. ระบบ normalize weight ให้รวมกันเป็น 1 เพื่อไม่ให้ scale เพี้ยน เหมาะกับการให้น้ำหนัก token/cost ต่างกัน เช่น cost สำคัญกว่า token

4) Threshold: เมื่อ score ≥ threshold request จะถูกปฏิเสธ ค่าเริ่มต้น 1.00 หมายถึง 100% ของ quota; ถ้าตั้ง 0.80 จะ block ตั้งแต่ใช้ถึงประมาณ 80%; ถ้าตั้ง 1.20 จะยอมให้เกิน quota ได้ประมาณ 20%

ตัวสร้างกฎ Signal

กำหนด keyword signal ภาษาไทย/อังกฤษเพื่อควบคุมการ route ได้จากหน้า admin

ใช้เป็น signal สำหรับ routing เท่านั้น: ถ้า prompt ยาวถึงค่านี้ router จะถือว่าอาจต้องใช้ premium reasoning ไม่ใช่ quota และไม่ตัดสิทธิ์ token usage โดยตรง

ไม่ชนกับ Quota Token

จำนวน prompt token สำหรับ Premium Routing ใช้เฉพาะตอนเลือกเส้นทางโมเดลใน Virtual Router โดยดู token ของ prompt/request เพื่อประเมินความซับซ้อน ส่วน Quota Token อยู่ใน Quota Policies และใช้ enforce ปริมาณ token ที่ผู้ใช้ใช้จริง/ถูก reserve ตาม daily/monthly limit. ดังนั้นสองค่านี้เป็นคนละชั้นกัน: routing decision vs quota enforcement.

Balanced models
ถ้า rule score ≥ threshold → เลือก candidate #3; ถ้าไม่ถึง → เลือก candidate #1
Code models
ถ้าเจอสัญญาณ coding → candidate #1; ถ้า premium score สูงหรือ LLM มองว่าซับซ้อน → candidate #3; ถ้าไม่เข้าเงื่อนไข → candidate #2
Long context
ถ้า prompt tokens ≥ long-context threshold หรือมีข้อความ ≥ 12 messages → candidate #1
